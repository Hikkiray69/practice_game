# DESIGN REWORK — Level / Characters / Fullscreen UI

## Цель

Перед дальнейшей визуальной и геймплейной полировкой зафиксировать новый дизайн-вектор проекта:

- явно проработать level design;
- явно проработать дизайн главного героя и NPC;
- перевести игру в полноэкранный формат;
- перестроить UI так, чтобы HUD был сбоку, а диалоги появлялись поверх игры снизу по центру;
- перевести взаимодействие с NPC с клика мышью на клавишу (`E`).

---

## 1. Level Design

### 1.1 Общая концепция сцены

Текущая сцена — технический прототип.  
Новая цель — сделать игровое пространство похожим на учебный 3D-хаб, а не на тестовую площадку.

### 1.2 Базовая структура уровня

Уровень для `MVP` должен содержать:

- центральную игровую зону;
- стартовую точку игрока;
- 3 смысловые зоны под миссии:
  - зона `Ответственность`,
  - зона `Прозрачность`,
  - зона `Скорость`;
- читаемые ориентиры в пространстве;
- NPC или терминалы взаимодействия в каждой зоне.

### 1.3 Принципы level design

- Игрок должен понимать, куда идти, без лишней путаницы.
- Важные объекты должны визуально выделяться.
- Пространство не должно быть пустым: нужны ориентиры, ритм, логика движения.
- Зона миссии должна визуально поддерживать ее смысл.

### 1.4 Визуальные ориентиры

Нужно использовать:

- световые акценты;
- цветовые коды зон;
- заметные интерактивные объекты;
- пол/дорожки/платформы, которые помогают навигации.

### 1.5 Практическое решение для первой итерации

Для следующего шага level design делаем не "огромный мир", а компактный хаб:

- одна основная арена;
- 3 выделенных сектора;
- минималистичный sci-fi / corporate training стиль;
- простые геометрические формы, но с читаемой композицией.

## 2. Дизайн главного героя

### 2.1 Роль героя

Главный герой — новый участник команды, проходящий вводную симуляцию.

### 2.2 Визуальный подход

На ближайшей итерации не нужен сложный персонаж-меш.  
Нужен stylized proxy-character, который уже читается как "игрок", а не как случайная сфера.

### 2.3 Требования к дизайну героя

- отдельная читаемая форма;
- выделение направления взгляда/переда;
- цвет, отличающий героя от NPC и окружения;
- визуальная простота, чтобы не грузить сцену.

### 2.4 Практический вариант для реализации

Первая версия:

- capsule / stylized humanoid silhouette;
- основной цвет: акцентный, но не конфликтующий с интерактивными зонами;
- легкая индикация "переда" персонажа;
- тень или подсветка под героем для читаемости позиции.

---

## 3. Дизайн NPC

### 3.1 Роль NPC

NPC — не просто объект активации, а источник контекста миссии.

### 3.2 Требования к визуальному различию

- NPC должны отличаться от героя формой и цветом;
- у NPC должна быть своя "роль" в кадре;
- интерактивный NPC должен быть заметен издалека;
- около NPC нужен понятный сигнал доступности взаимодействия.

### 3.3 Практический вариант для реализации

Первая версия:

- stylized персонаж/терминал/голограмма;
- уникальная цветовая подсветка;
- маркер взаимодействия над NPC;
- состояние:
  - далеко -> просто виден,
  - рядом -> появляется prompt на взаимодействие,
  - активен -> открывается dialogue overlay.

## 4. Новый UI/UX layout

### 4.1 Общий принцип

Игра должна ощущаться как полноценный игровой экран, а не как страница с блоками.

### 4.2 Fullscreen layout

Новая структура:

- игра занимает весь экран;
- 3D-сцена — фон и основной слой;
- HUD закреплен сбоку;
- диалоги и ответы появляются поверх сцены снизу по центру.

### 4.3 HUD

HUD должен быть:

- закреплен с одной стороны экрана;
- визуально цельным;
- стилизованным под игровой интерфейс;
- не мешающим обзору сцены.

HUD должен включать:

- текущую миссию;
- прогресс по миссиям;
- статус миссии;
- очки / качество / скорость;
- при необходимости таймер.

### 4.4 Блок диалога

Диалоги должны:

- появляться поверх сцены;
- быть расположены снизу по центру;
- содержать реплику NPC;
- показывать варианты ответа как игровые action-buttons;
- визуально отделяться от HUD.

### 4.5 Основные UX-правила

- HUD и dialogue overlay не должны конкурировать за внимание.
- Основной фокус игрока — сцена и взаимодействие.
- Интерфейс должен выглядеть "игровым", а не как админка/форма.

---

## 5. Новый способ взаимодействия

### 5.1 Отказ от клика мышью

Взаимодействие с NPC должно происходить не мышью, а клавишей действия.

### 5.2 Базовая схема

- движение: `WASD` / стрелки;
- взаимодействие: `E`;
- при приближении к NPC появляется prompt:
  - `Нажмите E для взаимодействия`.

### 5.3 Правила

- если игрок вне радиуса, `E` ничего не делает;
- если игрок в радиусе NPC, `E` открывает диалог;
- если диалог уже открыт, `E` не должен ломать состояние;
- управление должно работать независимо от раскладки.
