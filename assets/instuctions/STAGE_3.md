# STAGE 3 — UX/UI и геймплейные правила

## 3.1 Цель этапа

Зафиксировать UX-правила и пользовательские сценарии для `MVP-1`, чтобы следующий кодовый этап шел без двусмысленностей.

## 3.2 Flow экранов

Для проекта используется следующий flow:

1. `main-menu`
   - кнопка "Начать";
   - переход в `game`.
2. `game`
   - 3D-сцена;
   - HUD;
   - панель миссии/диалога;
   - выбор действий игрока.
3. `mission-result`
   - краткий итог выбора;
   - изменение метрик (очки/качество/скорость);
   - кнопка retry ("право на ошибку") или переход далее.
4. `profile/progress`
   - агрегированный прогресс игрока;
   - история миссии и текущие метрики.

Для `MVP-1` допустим единый экран с встроенным состоянием `mission-result` (без отдельного route), при сохранении той же логики flow.

## 3.3 Спецификация HUD

HUD обязан содержать:

- текущую миссию (название и/или идентификатор);
- статус миссии (`new`, `inProgress`, `completed`, `failed`);
- очки (`score`);
- скорость (`speed`);
- качество (`quality`);
- таймер (`MVP-2`, в `MVP-1` может быть скрыт/disabled).

Требования к отображению:

- значения обновляются сразу после выбора;
- статусы и метрики читаются без дополнительных кликов;
- при отсутствии данных используются безопасные значения по умолчанию (`0`/`new`).

## 3.4 Визуальные подсказки и фидбек

- Кнопки действий имеют явные подписи ("Начать миссию", варианты выбора, "Попробовать еще раз").
- После выбора игрок сразу видит текстовое последствие решения.
- Цвет/акцент состояния должен показывать переход миссии (`new -> inProgress -> completed`).
- Ошибки/некорректные действия не ломают flow: UI остается стабильным.

## 3.5 Право на ошибку (retry)

Правила retry:

- доступен после завершения миссии;
- сбрасывает выбранный вариант и последствия;
- возвращает миссию в `inProgress`;
- не вызывает критического штрафа и позволяет повторить выбор.

## 3.6 Начисление очков и прогресса (MVP-1)

- Каждому выбору соответствует фиксированное последствие:
  - `scoreDelta`
  - `qualityDelta`
  - `speedDelta`
- Метрики отображаются в HUD и служат базой для дальнейшей системы прогресса.
- Для `MVP-1` достаточно хранения в состоянии клиента; долговременное хранение переносится на поздние этапы.

## 3.7 Адаптивность и читабельность

- Desktop-first layout с двумя колонками: сцена + сайдбар.
- На узких экранах (tablet/mobile) переход в одну колонку.
- Текст и кнопки должны оставаться читаемыми без масштабирования страницы.

## 3.8 Факт соответствия текущей реализации

На текущем шаге в проекте `game/` уже реализованы:

- экран `game` с 3D-сценой;
- HUD со статусом и метриками;
- панель диалога с выбором;
- моментальный фидбек последствий;
- retry-механика по принципу "право на ошибку".

Остальные экранные маршруты (`main-menu`, `mission-result`, `profile/progress`) остаются частью UX-контракта и могут быть вынесены в отдельные route на следующем кодовом шаге.
